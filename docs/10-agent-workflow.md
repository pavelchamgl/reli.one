# 10. Agent Workflow

> Инструкции и контекст для AI-агентов (Cursor, Claude и др.), работающих с этим репозиторием.

## Структура проекта — быстрый ориентир

| Что нужно изменить | Где искать |
|--------------------|------------|
| API-эндпоинт | `backend/<app>/views.py` + `<app>/urls.py` |
| Модель БД | `backend/<app>/models.py` → миграция |
| Сериализатор | `backend/<app>/serializers.py` |
| Настройки Django | `backend/backend/settings.py` |
| Env-переменные | `envs/backend.env.example`, затем реальный `envs/backend.env` |
| API-запрос с фронта | `Frontend/Frontend3/src/api/` |
| Redux стейт | `Frontend/Frontend3/src/redux/` |
| UI-компонент магазина | `Frontend/Frontend3/src/` |

## Правила для агентов

### Нельзя без явного разрешения
- Изменять `docker-compose.yml` и `Dockerfile`
- Изменять `backend/backend/settings.py` (особенно security-настройки)
- Добавлять файлы в `envs/` (только `*.example`)
- Делать `git push` или `git push --force`
- Удалять или изменять миграции
- Изменять `nginx/default.conf`

### Обязательно перед коммитом
- Убедиться что в коде нет захардкоженных секретов, токенов, паролей
- Убедиться что в коде нет PII (email, телефон, адрес реального человека)
- Убедиться что новые env-переменные добавлены в `envs/backend.env.example`

### Стиль коммитов

Формат: `<type>(<scope>): <описание>`

Типы: `feat`, `fix`, `refactor`, `security`, `docs`, `chore`, `style`

Примеры из истории:
```
feat(accounts): make phone number optional in registration
fix(onboarding): align company account holder validation
security: remove secrets from repo, restrict CORS and ALLOWED_HOSTS
```

### Добавление новой интеграции
1. Добавить переменные в `envs/backend.env.example`
2. Добавить чтение через `os.getenv()` в `settings.py`
3. Задокументировать в `docs/06-integrations.md`

### Добавление нового Django-приложения
1. Создать через `manage.py startapp`
2. Добавить в `INSTALLED_APPS` в `settings.py`
3. Зарегистрировать URL в `backend/backend/urls.py`
4. Задокументировать в `docs/01-business-domains.md`

## Известные ловушки

- `STRIPE_SECRET_KEY_TEST` объявлен в `envs/backend.env`, но **не** читается в `settings.py` — при обращении к `settings.STRIPE_SECRET_KEY_TEST` будет `AttributeError`
- Миграции исключены из git — после изменения моделей нужно генерировать и применять миграции вручную
- `BaseURL` во фронте захардкожен как `"https://reli.one/api"` — для локальной разработки нужно явно переопределять
- `info.reli.one` проксирует `/admin` на backend — хост должен быть в `ALLOWED_HOSTS`

## Master Prompt для Cursor Agent

Ты работаешь как senior solution architect, backend lead, frontend lead, QA lead и product-minded technical lead для проекта reli.one.

Главная цель:
Стабилизировать проект перед масштабированием и перед внедрением AI-agent development workflow.

Правила:
1. Не меняй код без явной задачи.
2. Сначала анализируй и документируй.
3. Не меняй API-контракты без отдельного согласования.
4. Не меняй модели и миграции без migration strategy.
5. Payment/order/delivery/seller onboarding считаются критическими зонами.
6. Для критических зон сначала нужны regression tests.
7. Все изменения должны быть маленькими и reviewable.
8. Документацию обновляй вместе с архитектурными изменениями.
9. Не коммить секреты, env, private keys, certs.
10. Для каждого изменения показывай:
   - что изменено
   - почему
   - какие риски
   - какие тесты запускать
   - короткий commit message

## Когда использовать Master Prompt

Использовать:
- при старте большой архитектурной задачи
- перед рефакторингом payment/order/delivery
- перед изменением моделей БД
- перед задачами по CI/CD
- если агент начал предлагать слишком широкий или рискованный план

Не использовать:
- для маленьких точечных правок
- для простого review diff
- для исправления опечаток или мелкого UI

## Критические зоны проекта (повышенный риск)

Следующие области требуют особой осторожности:

- Payment (Stripe, PayPal, webhooks)
- Order lifecycle (создание, статусы, позиции)
- Delivery (расчет стоимости, PUDO, адреса)
- Seller onboarding (KYC, документы, валидация)
- Product / Variant (цены, параметры, наличие)

Правила работы с этими зонами:

- Любые изменения требуют проверки тестов
- Нельзя менять поведение без явного указания
- Перед изменениями нужно понять текущий flow
- Webhook-и должны быть идемпотентны
- Любые multi-step операции должны быть транзакционными

## Правила рефакторинга

- Не делать массовый рефакторинг сразу
- Рефакторинг выполняется маленькими шагами (1 логика = 1 изменение)
- Сначала фиксируется текущее поведение (тестами), потом меняется код
- Нельзя менять API-контракты без отдельного задания
- Нельзя менять модели без migration strategy
- После каждого изменения нужно проверить:
  - тесты
  - поведение API
  - интеграции

## Ожидаемый процесс работы агента

Агент должен работать по следующему циклу:

1. Понять задачу
2. Найти связанные файлы
3. Проанализировать текущую реализацию
4. Определить риски
5. При необходимости предложить план
6. Выполнить минимально необходимое изменение
7. Объяснить изменения
8. Указать какие тесты нужно запустить

Запрещено:
- Делать предположения без анализа кода
- Менять несколько доменов одновременно
- Делать "улучшения" без запроса
