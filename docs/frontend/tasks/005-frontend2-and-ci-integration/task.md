# FE-T005 — Frontend2 и интеграция тестов в CI

**Priority:** P1  
**Complexity:** Medium  
**Status:** To do

## Цель

Распространить подход **FE-T002** на **`Frontend/Frontend2`** (Vitest + RTL по необходимости) и **подключить прогон фронтовых тестов** к **GitHub Actions** (или актуальному CI репозитория) так, чтобы PR не ломали контур.

## Контекст

Два Vite-приложения должны иметь согласованные команды; без CI локальные тесты перестают запускаться.

## Scope

- Скопировать/адаптировать конфиг Vitest для Frontend2 (минимальный smoke + приоритетные юниты по матрице FE-T001).
- Обновить **workflow**: шаги `Frontend2` и `Frontend3` для `npm ci`, `npm run test`, при необходимости кэш зависимостей.
- Опционально: job **Playwright** с `if:` / расписанием после FE-T004.
- Обновить **[08. Testing strategy](../../../08-testing-strategy.md)** таблицу «Frontend» после появления скриптов `test` (фактическое состояние).

## Не входит в задачу

- Рефакторинг приложений вне необходимости для тестов.
- Удаление существующих backend-тестов из pipeline.

## Зависимости

- **FE-T002** (эталон конфига).
- **FE-T003** и **FE-T004** не блокируют старт FE-T005, но **зелёный CI** должен учитывать все уже добавленные тесты.

## Риски

- Удвоение времени CI — использовать кэш, параллельные job.
- Разные версии Node — зафиксировать в `.nvmrc` / `engines` если ещё не зафиксировано.

## Definition of Done

- [ ] Оба фронта имеют документированную команду тестов; она выполняется в CI.
- [ ] Падающий тест блокирует merge (если так принято в команде).
- [ ] Документ **08** или **testing-plan** отражает, что фронтовый раннер подключён (обновить формулировку «снимка»).
