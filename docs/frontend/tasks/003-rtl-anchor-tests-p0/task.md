# FE-T003 — Якорные RTL-тесты по сценариям P0

**Priority:** P0  
**Complexity:** Medium–High  
**Status:** To do

## Цель

Добавить **набор регрессионных тестов** (Testing Library) в **Frontend3** по строкам матрицы **FE-T001** с приоритетом **P0**: авторизация и защита маршрутов, критичные формы, базовая обработка ошибок API.

## Контекст

Backend уже проверяет бизнес-инварианты; фронт должен предотвращать регрессии **навигации**, **валидации**, **утечек отображения** (например неверный заказ) и **сломанного поиска** — в духе закрытых проблем **Task 007**.

## Scope

- Реализация тестов из утверждённой матрицы (ориентир **5–15** файлов на первую волну, не обязательно за один PR).
- Моки API согласно политике FE-T001; без реальных вызовов бэка в unit/RTL.
- Покрытие как минимум:
  - **ProtectedRoute** (неавторизованный пользователь не видит защищённый контент / редирект).
  - **Одна критичная форма** (например логин или фрагмент чекаута — по матрице).
  - **Обработка ошибки API** (тост или сообщение пользователю — по фактической реализации).
- Обновление матрицы: отметить выполненные строки или ссылки на файлы тестов.

## Не входит в задачу

- Полное покрытие всего каталога MUI-компонентов.
- E2E (FE-T004).
- Изменение бизнес-логики без отдельной задачи (тесты должны фиксировать **текущее** согласованное поведение).

## Зависимости

- **FE-T002** (раннер и `renderWithProviders`).

## Риски

- Хрупкие селекторы — приоритет **ролям** и **д accessible name** из Testing Library.
- Флаки из-за `waitFor` без явного условия — избегать голых таймаутов.

## Definition of Done

- [ ] Для каждого целевого P0-сценария из матрицы есть тест или явно задокументированное исключение (почему отложено).
- [ ] Прогон `npm run test` в Frontend3 зелёный в CI (после подключения job в FE-T005 или в этом PR, если CI уже обновлён).
- [ ] Нет секретов и персональных данных в фикстурах.
