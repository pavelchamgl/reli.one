# FE-T001 — Матрица тестовых сценариев и конвенции

**Priority:** P0  
**Complexity:** Low  
**Status:** To do

## Цель

Зафиксировать **что** тестируем на фронте в первую очередь и **как** оформляем тесты (имена файлов, размещение, хелперы, политика моков), чтобы пилот Vitest (FE-T002) не расходился со стилем репозитория.

## Контекст

Без общей матрицы и конвенций разные разработчики выберут несовместимые подходы к мокам HTTP и обёртке Redux/Router, что усложнит поддержку и CI.

## Scope

- Таблица **сценарий → приоритет (P0/P1) → уровень (unit / RTL / e2e)** для **Frontend3** и отдельно краткий блок для **Frontend2**.
- Соглашения: суффиксы файлов, расположение рядом с кодом vs `__tests__`, язык описаний `it`/`test`.
- Описание целевого хелпера **`renderWithProviders`** (какие провайдеры обязательны: Redux, Router, i18n).
- Выбор **основного** способа моков HTTP на первом этапе: MSW vs `vi.mock` слоя API — зафиксировать решение и исключения.
- Политика для **Sentry** и **OAuth-виджетов** в тестах (заглушки).

## Не входит в задачу

- Установка npm-зависимостей и конфигов Vitest (это FE-T002).
- Написание самих тестов (FE-T003 и далее).

## Зависимости

- Согласованный **[план тестирования](../../testing-plan.md)** и P0 из **[08. Testing strategy](../../../08-testing-strategy.md)**.

## Риски
