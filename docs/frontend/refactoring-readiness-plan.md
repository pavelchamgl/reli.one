## Принцип фиксации изменений

1. **Один PR — одна законченная единица работы**: тесты + при необходимости правка только того кода, без которого тесты не зелёные; либо только документация/конфиг.
2. **После merge — обновить** в том же PR (или сразу следующим микро-PR): строки в [test-matrix.md](./test-matrix.md), чеклисты в [tasks/003-rtl-anchor-tests-p0/task.md](./tasks/003-rtl-anchor-tests-p0/task.md), при смене статуса задачи — [tasks/README.md](./tasks/README.md).
3. **Сообщения коммитов** — коротко, без служебных трейлеров; формат см. в [10-agent-workflow.md](../10-agent-workflow.md).
4. **Перед рефакторингом «сдвиг слоёв»** (router, store, API-слой) — в зоне изменения должны быть **якорные тесты** или явное решение «отложено» в матрице с краткой причиной.

## Фаза 0 — Стабилизация контекста (короткая)

Цель: не начинать рефактор без ясной базовой линии.

| Шаг | Действие | Фиксация (пример PR) | Критерий готовности |
|-----|----------|----------------------|---------------------|
| 0.1 | Убедиться, что **`frontend3`** и **`e2e_frontend3`** в CI зелёные на `main` | При необходимости — отдельный `fix` | Зелёный pipeline на целевой ветке |
| 0.2 | Локально: `npm run test` в `Frontend/Frontend3`; при работе над e2e — `npm run build && npm run test:e2e` | — | Без регрессий перед фазой 1 |
| 0.3 | **Frontend2:** job **`frontend2`** сейчас часто падает на **ESLint** (см. [README](./README.md)) | Не смешивать с рефактором Frontend3; закрывать отдельной дорожкой **FE-T005** (чанками: baseline eslint-disable, правки правил, или поэтапное исправление файлов) | Понятная запись в задаче FE-T005 / debt, если полный зелёный lint откладывается |

## Фаза 1 — Закрытие **FE-T003** (якорные RTL по матрице P0 в Frontend3)

Связь: **FE-T003** — основная «открытая» задача по тестам; [**test-matrix.md**](./test-matrix.md) — источник правды по строкам.

Рекомендуемый **порядок PR** (от более изолированного к более связному):

| № | Тема PR | Содержание | После merge |
|---|---------|------------|-------------|
| 1.1 | **Логин / регистрация (Yup)** | RTL: валидация полей (ошибки/успех), без реального OAuth/бэка; моки API при сабмите | Строка в матрице → «Покрыто» + путь к `*.test.jsx` |
| 1.2 | **Ошибки API / retry, тосты** | RTL или тонкий интеграционный тест на реальный путь «запрос упал → пользователь видит сообщение» при текущей реализации | То же |
| 1.3 | **Корзина / чекаут** | Начать с **RTL** на срез состояния (отображение позиций, блокирующие ошибки); при необходимости **отдельный PR** — минимальное расширение **e2e** только если без него нельзя зафиксировать критичный happy path | Обновить матрицу; не раздувать e2e без нужды |

**Закрытие FE-T003 как задачи:**

- В [test-matrix.md](./test-matrix.md): все строки **P0** для Frontend3 заполнены тестами **или** помечены осознанным **Backlog / Отложено** с одной строкой обоснования в ячейке или сноске под таблицей.
- В [tasks/003-rtl-anchor-tests-p0/task.md](./tasks/003-rtl-anchor-tests-p0/task.md): **Definition of Done** согласован с матрицей.
- В [tasks/README.md](./tasks/README.md): статус **FE-T003** → **Done**, когда DoD выполнен.

**Параллельно (не блокер для 1.1–1.3, но желательно до фазы 3):**

- **FE-T001** — формальное ревью матрицы/конвенций с владельцем продукта (артефакт уже есть; статус в README при необходимости уточнить).
- **FE-T005** — приоритизировать «зелёный lint + test в CI» для Frontend2 **отдельными PR**, не смешивая с рефактором магазина.

## Фаза 2 — Критерии готовности к **рефакторингу Frontend3**

Рефакторинг здесь означает **структурные изменения**: разбиение крупных экранов, перенос логики из компонентов, изменение границ `redux`-слайсов, контрактов с `src/api`, схемы роутинга — без изменения продуктовых правил, если отдельно не согласовано.

**Гейты (минимум перед первым крупным PR):**

| # | Гейт | Проверка |
|---|------|----------|
| G1 | **Матрица P0** | Все неотложенные P0 в Frontend3 покрыты тест **или** явно отложены |
| G2 | **CI** | `frontend3` + `e2e_frontend3` зелёные |
| G3 | **Карта зон риска** | Краткий список модулей под рефактор (файлы/папки) согласован или записан в задаче/issue |
| G4 | **Обёртка тестов** | Для экранов с **i18n** в тестах используется согласованное расширение `renderWithProviders` (см. [test-matrix.md](./test-matrix.md)) — чтобы рефактор не ломал все тесты из-за отсутствия провайдера |

**Опционально (по нагрузке команды):**

- Пилот **MSW** или единый слой моков HTTP — отдельная задача; не блокирует старт рефактора, если текущие `vi.mock` достаточны.

## Фаза 3 — Пошаговый **рефакторинг Frontend3** (малые PR)

Рекомендуемый порядок (каждый пункт — **1 или несколько узких PR**):

1. **Утилиты и чистые функции** без UI — почти без регрессий в RTL.
2. **Один слайс / один домен Redux** — с сохранением публичных селекторов или с миграцией в том же PR; тесты на редьюсеры/экшены при наличии нетривиальной логики.
3. **API-слой (`src/api`)** — вынос общих хелперов; не менять контракт запросов/ответов без согласования ([000-project-core](../../.cursor/rules/000-project-core.mdc)).
4. **Компоненты снизу вверх** — листья дерева → контейнеры; после каждого значимого шага: `npm run test`, при затрагивании навигации — выборочно e2e.
5. **Роутинг и layout** — только когда G1–G4 выполнены и есть тесты на ключевые маршруты (в т.ч. уже существующие **ProtectedRoute**).

**Запрещённая практика:** один огромный PR «переписали половину приложения» без промежуточной зелёной CI.
