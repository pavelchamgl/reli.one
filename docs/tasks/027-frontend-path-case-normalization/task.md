# Task 027 — Нормализация регистра путей каталога компонентов (Frontend3)

**Priority:** P2
**Complexity:** Medium (риск выше из-за case-insensitive FS)
**Status:** Done (repo-scope, Вариант 2 — заглавный `Components/`)

## Цель

Устранить «регистровый долг» во Frontend3: в git-индексе сосуществуют два
регистра одного и того же верхнего каталога — `Components/` и `components/`.
Привести к единому регистру, убрать риск дублей файлов и поломок на
case-sensitive системах (CI/Linux/Docker), после чего снять временный
pre-commit хук.

## Контекст

- macOS — **case-insensitive** ФС: физически существует один каталог
  `Frontend/Frontend3/src/Components/` (с заглавной). Git-индекс —
  **case-sensitive**, поэтому хранит пути в двух пространствах имён.
- Текущее состояние индекса (на коммите `4076b528`):
  - под **`Components/`** (заглавная) — **341** файл (основная масса приложения);
  - под **`components/`** (строчная) — **58** файлов:
    - `components/seller/onboarding/` — 43,
    - `components/ui/` — 15 (shadcn-компоненты).
  - регистровых дублей «один-в-один» сейчас **0**.
- Импорты ссылаются на оба регистра: строчный `/components/...` —
  ~56 вхождений в 32 файлах; заглавный `/Components/...` — в 59 файлах.
- Есть временный pre-commit хук `.git/hooks/pre-commit`, запрещающий коммиты в
  `Frontend/Frontend3/src/Components/Seller/onboarding/views/` (он заставляет
  держать onboarding-views в строчном регистре).

### Чем это опасно
- На case-sensitive системе строчные и заглавные пути станут **разными**
  каталогами → дублирование компонентов, рассинхрон импортов, падение
  сборки/тестов.
- Любой `git add --renormalize .` или массовая операция повторно плодит
  регистровые дубли (уже наступали на это — см. ниже «История»).

### История (как всплыло)
- Task 026 (forward-only EOL, коммит `4076b528`) — корректен, сохранён.
- Попытка полной нормализации (`git add --renormalize .`, коммит `9622c956`)
  создала 27 дублей под `Components/...` и была откатана
  `git reset --hard HEAD~1`. Откат из-за case-insensitive FS физически удалил
  27 файлов; восстановлены `git checkout HEAD -- ...`. Коммит `9622c956`
  остаётся в `git reflog` (anchor `9622c95687b1c6ad6d525eb6a822cb4310261f4b`).

## Scope (область)

- Каталог `Frontend/Frontend3/src/` — приведение `Components/` ↔ `components/`
  к единому регистру.
- Обновление импортов под выбранный регистр.
- Снятие/упрощение pre-commit хука после устранения долга.

## Не входит в задачу

- Изменение логики компонентов, рефакторинг кода.
- EOL-нормализация (закрыта Task 026).
- Перемещение/переименование компонентов по смыслу (только регистр путей).

## Зависимости

- Task 026 (EOL policy) — выполнена.

## Риски

- **Case-only rename на macOS** не работает напрямую: нужно переименование
  через промежуточное имя (`git mv A tmp && git mv tmp a`) или операции с
  `core.ignorecase=false`.
- Пропущенный импорт → битый бандл. Обязательна полная проверка ссылок и сборки.
- Возможны ссылки на регистр в конфигурации (alias, jsconfig/vite, тесты,
  Code Connect/figma, storybook).

# Iterations

## Iteration 1 — Decision: целевой регистр

### Цель
Выбрать один регистр верхнего каталога компонентов.

### Варианты
- **Вариант A — строчный `components/` (рекомендуется).**
  Совпадает с конвенцией shadcn (`components/ui`) и с намерением хука
  (onboarding-views в строчном). Минус: переименовать больше файлов (341).
- **Вариант B — заглавный `Components/`.**
  Меньше файлов переименовывать (58), но противоречит shadcn/хуку; `ui/`
  обычно строчный, появится несогласованность.

### Решение
- **Выбран Вариант B — заглавный `Components/`** (по согласованию с владельцем).
  Причина: при инвентаризации выяснилось, что регистр различается не только на
  верхнем уровне, но и в подпапках (`Seller` 154, `Product` 50, `Payment` 28,
  `ProfileNav`, `Basket`, `Orders`, `Catalog`, `Auth` и т.д. — все CamelCase),
  тогда как строчными были только `components/seller/onboarding` (43) и
  `components/ui` (15). Полный переход на строчный регистр потребовал бы
  переименования всего дерева компонентов и почти всех импортов приложения
  (сотни файлов) — слишком большой и рискованный единичный diff. Поэтому
  консолидировали меньшинство (58 файлов) к доминирующей CamelCase-конвенции.
- Алиасы shadcn в `components.json` обновлены на `@/Components` и `@/Components/ui`.

### Статус
- [x] Decision recorded

## Iteration 3 — Переименование путей (через staging-имя)

### Цель
Безопасно сменить регистр на case-insensitive FS.

### Действия (пример для перехода к строчному)
- Через промежуточное имя, чтобы git зафиксировал смену регистра:
  ```sh
  git mv Frontend/Frontend3/src/Components __tmp_components
  git mv __tmp_components Frontend/Frontend3/src/components
  ```
  либо выполнять на checkout с `core.ignorecase=false`.
- Слить ранее раздельные `components/` (58) и `Components/` (341) в один
  регистр без потери файлов.
- Не менять содержимое файлов на этом шаге.

### Ограничения
- Только переименование путей/регистра, без правок кода.

### Статус
- [x] Rename done — физические файлы уже лежали под `Components/Seller/onboarding`
      и `Components/ui`, поэтому перемещения не требовались: 58 записей индекса
      перепривязаны к заглавному регистру через `git mv` (case-only, статус `R`).
