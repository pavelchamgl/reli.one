## Контекст

Загрузка изображений товара продавцом сейчас устроена так:

- frontend кодирует все файлы в base64 и отправляет **одним** запросом `POST /api/sellers/products/{id}/images/bulk_upload/`;
- backend в цикле создаёт `BaseProductImage`, и в `BaseProductImage.save()` **синхронно** выполняется PIL: resize/pad → WebP 1000×1000;
- нет лимита на количество изображений и на размер одного файла (лимит `MAX_UPLOAD_SIZE = 13MB` применяется только к license-файлам);
- публичные сериализаторы отдают все `images[]` без учёта статуса обработки.

### Проблемы

1. **Транспорт.** base64 даёт +~33% к размеру; все файлы в одном JSON. При 15–20 фото пробивается `nginx client_max_body_size 15M` и дефолтный `DATA_UPLOAD_MAX_MEMORY_SIZE` (2.5MB) Django.
2. **Синхронный PIL в HTTP-запросе.** Тяжёлые файлы (до 13MB) занимают gunicorn-воркер на сотни мс — секунды, риск таймаутов прокси, деградация под пиками загрузок.
3. **Отсутствие лимитов** делает поведение непредсказуемым (память вкладки, диск, время запроса).
4. **Partial success.** Product создаётся отдельным запросом раньше, чем images; падение шага images оставляет товар без фото без понятной обратной связи.

### Ограничения инфраструктуры (важно для выбора решения)

- Стек: **PostgreSQL 17 + gunicorn (4 WSGI-воркера)**.
- **Нет** Celery, Redis, иных брокеров и воркеров. Кэш Django — **DB-backed** (`django_cache` таблица), Redis в проекте отсутствует.
- Медиа — локальный `MEDIA_ROOT`, шарится между сервисами через volume.
- В `ProductMedia` уже есть поле `status` (модель частично готова к статусной модели).
- В `requirements.txt` присутствует `cloudinary`, но для изображений товара не используется.

Добавление async = введение нового класса инфраструктуры, которого в проекте нет. Решение обязано не противоречить целевой архитектуре (modular monolith, без микросервисов и event-driven сложности без явного запроса — см. `050-architecture.mdc`).

## Решение

Ввести **асинхронную обработку изображений на статусной модели**, реализованную **без брокера** — через периодически запускаемую management-команду (cron / loop-контейнер), которая обрабатывает записи в статусе `pending`.

### Ключевые принципы

1. **Async-shaped контракт с первого дня.**
   Endpoint загрузки сразу отвечает `201 { id, status: "pending", image_url: null }`. Frontend показывает статус слота и поллит до `ready`. Реальный движок обработки (cron сейчас, очередь позже) — деталь реализации, контракт не меняется.

2. **Транспорт — `multipart/form-data`, по одному файлу.**
   Новый endpoint `POST /api/sellers/products/{id}/images/`. Это снимает проблему суммарного размера: в лимиты инфраструктуры вписывается **один** файл, а не пачка. Legacy base64 `bulk_upload` сохраняется для обратной совместимости.

3. **Статусная модель.** `pending → processing → ready | failed`. Claim задач — через `select_for_update(skip_locked=True)`, чтобы перекрывающиеся запуски команды не брали одну запись дважды. Retry для `failed` по счётчику `attempts < 3`.

4. **Алгоритм WebP не меняется.** `resize_and_pad`/`process_image` (WebP 1000×1000, белый фон) переносятся в обработчик без изменения логики.

5. **Аддитивность и отсутствие breaking changes.** Новые поля модели имеют дефолты, при которых существующие строки и legacy-путь остаются валидными; публичный контракт `images[]` сохраняется, но дополняется фильтрацией только `ready`.

### Выбранный воркер: cron + management-команда

Команда `process_pending_product_images --batch N`:
- идемпотентна, безопасна при параллельных запусках (`skip_locked`);
- запускается loop-контейнером `image_worker` в `docker-compose` (тот же образ backend, тот же media-volume);
- логирует результат каждой записи; ошибки фиксируются в `processing_error`.

**Почему так:** durable (очередь живёт в Postgres), есть retry, ноль новой инфраструктуры кроме самого процесса-воркера, тривиальный деплой и откат. Дизайн команды (claim → process → finalize) совместим с последующей заменой на django-q2/Procrastinate без изменения контракта и фронта.

## Рассмотренные альтернативы

| Вариант | Новая инфра | Durable | Retry | Латентность | Вердикт |
|---|---|---|---|---|---|
| `on_commit` + ThreadPool | нет | ✗ (умирает с воркером) | ✗ | мгновенно | Отклонено: не «железобетонно» |
| **Cron + management-команда** | loop-контейнер | ✓ (БД) | ✓ | сек | **Выбрано** |
| django-q2 (ORM-брокер) | worker-сервис | ✓ | ✓ | сек | Целевой апгрейд (Фаза 3) |
| Procrastinate (LISTEN/NOTIFY) | worker-сервис | ✓ | ✓ | низкая | Альтернатива апгрейда |
| Celery + Redis | Redis + worker + monitoring | ✓ | ✓ | низкая | Отклонено: избыточно для ресайза |
| Cloudinary (offload) | внешний сервис | ✓ | ✓ | — | Отклонено: смена storage-модели |

Транспорт base64-bulk (сохранить, поднять лимиты) отклонён: лечит симптом, не решает память/время/partial-failure и блокирует будущие фичи (reorder/main).

## Последствия

### Положительные
- HTTP-запрос быстрый и предсказуемый; gunicorn-воркеры не заняты PIL.
- Устойчивость к файлам до 13MB и к пикам загрузок; per-image retry.
- Контракт финальный — Фаза 2 (DnD/paste/lightbox/main/reorder) и Фаза 3 (очередь, CDN) не ломают фронт.
- Публичная витрина не показывает необработанные изображения (фильтр `ready`).

### Отрицательные / цена
- Новый ops-surface: сервис `image_worker` нужно деплоить и мониторить.
- Усложнение фронта: состояние `processing` + polling вместо «сразу картинка».
- Миграция + временное хранение оригинала (диск, последующая очистка).
- Больше тестов и сценариев отказа.

### Нейтральные / отложенные
- Переход cron → django-q2/Procrastinate при росте нагрузки (Фаза 3).
- Дополнительные размеры (thumbnail/zoom-derivative) и CDN — вне scope этого ADR.

## Инварианты совместимости (обязательны)

1. Публичный ответ `BaseProductDetailSerializer.images` остаётся прежней формы; добавления полей — только аддитивные.
2. Публичная витрина и `get_product_cover_image` используют только изображения в статусе `ready`; главное фото остаётся детерминированным (см. `iteration-7-7`).
3. `related_name='images'` на `BaseProduct` сохраняется.
4. Legacy `POST .../images/bulk_upload/` (base64) продолжает работать без изменения поведения.
5. Не затрагиваются flow order / payment / delivery.

## Обратимость (по ADR-08)

- Миграция аддитивна: новые поля (`status` default `ready`, `original_upload` nullable, `processing_error` blank, `attempts` default 0). Откат миграции возможен; существующие данные не модифицируются.
- Откат фичи целиком: отключить `image_worker`, фронт вернуть на legacy `bulk_upload`. Данные `ready` остаются валидными.

## Метрики для решения о Фазе 3 (cron → очередь)

- Среднее/95p время обработки одного изображения.
- Глубина очереди `pending` и время ожидания.
- Доля `failed` и исчерпанных retry.
- Занятость воркера и частота пустых проходов команды.
